## 三轨文档体系

项目采用三轨文档体系，每轨有不同的真相源和维护时机：

| 轨道 | 标记 | 位置 | 真相源 | 对齐方向 | 维护时机 |
|------|------|------|--------|---------|---------|
| 🔴 Intent | 冻结 | OpenSpec（不在 docs/ 下，见下方 [OpenSpec Intent 轨](#openspec-intent-轨)） | 人类意图 | 冻结 | 编码前创建后冻结 |
| 🟢 Contract | 跟随代码 | docs/architecture/ + docs/modules/ | 代码 | 代码→文档 | 代码变更时更新 |
| 🔵 Constraint | 驱动代码 | AGENTS.md + docs/conventions/ | 团队共识 | 文档→代码 | 约束变更时更新 |

**三轨的区别与适用场景：**

- **🔴 Intent 轨**：记录"为什么要做"和"打算怎么做"。在编码之前创建，编码开始后冻结不再修改。**Intent 轨文档不在 docs/ 目录下**，而是位于独立的 `openspec/` 目录——这是因为它的真相源（人类意图）和维护规则（冻结不变）与 docs/ 下所有文档完全不同。适用于 OpenSpec 工作流中的 proposal、design、tasks、spec 等制品。当实现偏离 Intent 时，应通过新的 OpenSpec 变更流程处理，而非直接修改 Intent 文档。详见下方 [OpenSpec Intent 轨](#openspec-intent-轨) 章节。
- **🟢 Contract 轨**：记录"代码当前是什么样"。真相源是代码本身，文档必须 100% 反映代码现状。适用于架构文档、模块文档等描述性文档。代码变更时必须同步更新 Contract 文档。
- **🔵 Constraint 轨**：记录"代码应该遵守什么规则"。真相源是团队共识，文档驱动代码行为。适用于 AGENTS.md 中的编码规则、conventions/ 下的编码规范等。规则变更时更新文档，新代码必须遵守现有约束。

## 文档编写原则

本文档体系遵循以下 7 个编写原则：

### 1. 职责清晰

每份文档有且仅有一个职责，能用一句话概括"这份文档回答什么问题"。没有两份文档回答同一个问题——如果内容需要引用其他文档的信息，不复制，用链接。

### 2. 简要精准

- 面向 AI 的文档（AGENTS.md、conventions/）：精确规则，可执行步骤，步骤可闭环验证
- 面向人类的文档（architecture/、modules/）：段落说明 + Mermaid 图，充足的上下文和背景，场景驱动有头有尾

### 3. 层次分明

章节层级不超过 L3（`# → ## → ###`），每个 L2 章节聚焦一个主题，L3 用于细分。避免"所有内容堆在一个大章节下"。

### 4. 逻辑自洽

文档内容不与其他文档矛盾。Contract 轨文档之间如有重叠（如 modules/client-auth.md 和 architecture/request-lifecycle.md 都涉及认证），通过"相关文档"链接交叉引用，而非重复描述。

### 5. 科学有据

设计决策有参考来源（学术研究、业界标准、项目实践经验）。引用强度采用 RFC 2119 三级体系（MUST/SHOULD/MAY），经过 25+ 年工程验证。

### 6. 双范式写作

| 维度 | 面向 AI 的文档 | 面向人类的文档 |
|------|---------------|---------------|
| 适用范围 | AGENTS.md、conventions/ | architecture/、modules/ |
| 写作要求 | 精确规则，无歧义 | 充足上下文，场景驱动 |
| 格式偏好 | 步骤列表、代码示例、检查清单 | Mermaid 图、段落说明、对比表格 |
| 验证标准 | 步骤可闭环执行 | 读者可理解来龙去脉 |
| 典型读者 | AI 编码助手 | 开发者、维护者、新人 |

### 7. 索引具体规则泛化

- **索引性内容**（文档导航、文件列表）必须精确到具体文件路径——索引是当前事实的快照，需要精确
- **规则性内容**（维护策略、反模式、对齐机制）使用泛化描述，不绑死具体文件名——规则是未来也适用的指导，需要稳定

## 引用强度说明

本文档体系采用 RFC 2119 三级引用强度，以明确规则的约束力：

| 级别 | 关键词 | 含义 |
|------|--------|------|
| ⛔ MUST | MUST / SHALL | 绝对要求，无例外 |
| ⚠️ SHOULD | SHOULD / RECOMMENDED | 推荐，允许有充分理由的例外 |
| 💡 MAY | MAY / OPTIONAL | 完全可选 |

## 文档系统设计

### 设计初衷

本项目的文档体系是为了解决三个核心痛点而设计的：

**痛点一：文档职责混乱**

在重新设计之前，项目文档存在严重的职责交叉：AGENTS.md（259 行）与 docs/ 大量内容重复，AI 上下文过载（ETH Zurich 研究表明过长的 AGENTS.md 反而降低 AI 性能）。docs/ 按"文档类型"组织（教程/指南/参考），而非按"模块"组织，人类难以一眼看懂代码的运行逻辑与业务逻辑。

**痛点二：缺少 Contract 文档**

OpenSpec 的 spec.md 是设计意图（Intent），归档时冻结不变。项目缺少 100% 反映代码现状的"合同文档"——人类可读、AI 可引用的代码映射文档。学术论文研究表明，64% 的开源项目包含过时文档，代码-注释不一致与 bug 引入有显著正相关。

**痛点三：缺少文档系统说明**

没有标准化的文档编写模板，没有说明文档系统的指引文档，新人文档编写无规范可循。不同的文档风格差异大，维护成本高。

### 设计理念

本文档体系的核心设计理念可以用一句话概括：

> **让每份文档有且仅有一个真相源，让每份文档有且仅有一个职责。**

具体来说：

- **真相源唯一**：Intent 的真相源是人类意图，Contract 的真相源是代码，Constraint 的真相源是团队共识。一份文档不试图同时服务两个真相源。
- **职责内聚**：每份文档必须能用一句话概括其职责，且没有两份文档回答同一个问题。
- **读者导向**：文档不是写给机器的，是写给人的（包括未来的自己、AI 编码助手、新加入的团队成员）。不同读者在不同场景下需要不同类型的文档。
- **最小维护成本**：文档组织方式应使得"代码变更时只需要更新对应的文档"，而非"代码变更时需要到处更新"。

**关键设计约束：**

- AGENTS.md 是 AI 编码助手的入口文件，保持精简（约束摘要 + 引用索引），详细内容委托给 docs/
- docs/ 是人类的完整知识库，按"模块+关注点"组织
- OpenSpec 是设计决策的历史记录，编码开始后冻结不变

### OpenSpec Intent 轨

> 🔴 Intent 轨文档**不在 docs/ 目录下**，而是位于独立的 `openspec/` 目录。这是因为它的真相源（人类意图）和维护规则（冻结不变）与 docs/ 下所有文档完全不同——docs/ 是 Contract 轨（跟随代码变化），而 Intent 轨在编码开始后冻结。

**为什么需要独立的 Intent 轨？**

在传统的文档体系中，设计文档往往与开发文档混在一起。问题是：设计文档记录的是"最初打算怎么做"，而代码实现可能因为各种原因偏离设计。如果设计文档跟随代码变化，就丢失了决策的原始上下文；如果不跟随变化，又与代码现状不一致。Intent 轨通过"冻结"策略彻底解决了这个矛盾——它不试图与代码保持一致，而是忠实地记录编码前的设计意图。

**OpenSpec 的位置与结构**

```
openspec/
├── specs/                  ← Intent 轨文档（冻结），每个 capability 一个目录，含 spec.md
└── changes/                ← 变更流程制品（活跃中为 Intent，归档后冻结）
    └── <change-name>/
        ├── proposal.md     [为什么做 + 做什么]
        ├── design.md       [技术设计决策]
        ├── tasks.md        [实施任务列表]
        └── specs/          [各能力的 spec.md]
```

当前有 17 个 capability 的 spec.md。完整的 capability 列表和路径见 [AGENTS.md 中的 OpenSpec Intent 索引](../AGENTS.md)。

**OpenSpec 变更工作流**

```
explore（探索）→ new（创建变更）→ continue（逐步制品化）→ apply（实施）→ archive（归档冻结）
```

| 阶段 | 产出 | 说明 |
|------|------|------|
| explore | 讨论记录 | 探索问题、澄清需求 |
| new → continue | proposal → design → specs → tasks | 结构化的变更制品 |
| apply | 代码/文档变更 | 按 tasks.md 执行实施 |
| archive | 冻结制品 | 变更完成，所有制品冻结归档 |

**Intent 轨与 Contract 轨的关系**

Intent 和 Contract 是互补的，不是替代的：

| 维度 | 🔴 Intent 轨 | 🟢 Contract 轨 |
|------|-------------|---------------|
| 回答的问题 | "最初打算怎么做" | "代码现在是什么样" |
| 真相源 | 人类意图 | 代码 |
| 时间性 | 冻结（编码前快照） | 实时（跟随代码变化） |
| 读者场景 | 理解设计决策的来龙去脉 | 理解代码的当前行为 |
| 典型读者 | 维护者 / 未来自己 / 新人 | 开发者 / AI Agent |

**当实现偏离 Intent 时**：不应修改 Intent 文档（那样会丢失原始设计意图），而应通过新的 OpenSpec 变更流程记录偏差的原因和新的设计决策。这样，Intent 轨形成了一条完整的"设计决策时间线"。

### 三轨分类标准

为什么选择"三轨"而非"单轨"或"两轨"？

**单轨方案的失败**：将所有内容放 AGENTS.md。问题：AGENTS.md 过长降低 AI 性能（ETH Zurich 研究），且无法同时服务"冻结的设计意图"和"实时更新的代码映射"两种矛盾需求。

**两轨方案的不足**：将文档分为"设计文档"和"开发文档"。问题：无法区分"描述代码现状的文档"（Contract）和"约束代码行为的文档"（Constraint）——前者跟随代码变化，后者驱动代码变化，维护时机和方向完全不同。

**三轨方案的合理性**：

| 区分维度 | Intent vs Contract | Contract vs Constraint |
|---------|-------------------|----------------------|
| 核心区别 | 时间维度：编码前 vs 编码后 | 方向维度：代码→文档 vs 文档→代码 |
| 真相源 | 人类意图 vs 代码 | 代码 vs 团队共识 |
| 变更频率 | 冻结不变 | 频繁（跟随代码） | 中等（规则变更时） |
| 读者场景 | "为什么这样设计" | "代码现在是什么样" | "代码应该怎么写" |

### docs/ 目录的组织方式

docs/ 按三大分组组织，每个分组有明确的切分标准：

```
docs/
├── architecture/    ← 按"读者的问题"切分（每个文件回答一个架构问题）
├── conventions/     ← 按"关注点"切分（每个文件聚焦一个编码关注点）
└── modules/         ← 按"模块"切分（每个文件描述一个业务/技术模块）
```

**为什么按"模块+关注点"而非按 Diátaxis 四象限组织？**

Diátaxis（教程/指南/参考/解释）是优秀的文档分类法，但它是面向"产品用户"的。脚手架项目的读者是"开发者/维护者"，他们需要的是：
- 理解架构（→ architecture/）
- 遵守规则（→ conventions/）
- 了解某个模块怎么用（→ modules/）

按 Diátaxis 组织会导致信息分散：了解 auth 模块需要同时看"教程"和"参考"两个文件。按模块组织则保证"一个模块 = 一个文件 = 所有相关信息集中"。

**为什么 business 模块和 technical client 放在同一个 modules/ 目录？**

虽然业务模块（auth、system-config）和技术客户端（client-*）在概念上不同，但它们的使用场景是一致的：开发者需要了解某个模块的功能、设计、API 和配置。将它们放在一起减少了认知负担，且 modules/README.md 中的分类表格已经做了业务/技术区分。

### 模板系统的设计哲学

**为什么需要模板？**

1. **降低编写门槛**：新人无需从零思考文档结构，直接按模板填充
2. **保证结构一致性**：同类文档有相同的章节骨架，便于快速定位信息
3. **减少维护遗漏**：模板的检查清单（checklist）确保关键章节不被遗忘

**为什么采用"固定层 + 半固定层"而非完全固定？**

完全固定的模板太僵硬——不同的架构文档回答不同的问题（系统全景 vs 设计模式），强制统一章节名会降低文档的表达力。完全灵活则失去模板的意义——不同人写出完全不同结构的文档。

双层结构是折中方案：
- **固定层**（外壳）：标题、📋 目录、必备章节（概述/相关文档/变更历史）→ 保证基本一致性
- **半固定层**（内容）：章节名根据实际命名、章节可选（无内容标注"本节无内容"）→ 保证灵活性

**三种模板的差异理由**：

| 模板 | 统一程度 | 理由 |
|------|---------|------|
| architecture/ | 每文件独立结构 | 每个文件回答不同的架构问题（C4 图 vs 时序图 vs 类图），强行统一章节名反而降低表达力 |
| conventions/ | 统一外壳 | 所有规范文档的职责相同（列出规则+示例），统一外壳降低认知负担 |
| modules/ | 统一 8 章节 | 所有模块文档的读者期望相同（先了解模块→再看怎么用→查 API→查配置→查相关文档→查变更历史），统一章节顺序便于快速定位 |

### 引用强度体系

**为什么选择 RFC 2119 三级体系？**

RFC 2119 是经过 25+ 年互联网工程验证的语义标准（MUST/SHOULD/MAY）。超过 3 级的体系（如 4 级：必须/强烈建议/建议/可参考）会增加认知负担，且无有效性证据支撑。三级恰好对应：

- ⛔ **MUST**：违反会导致 bug 或架构腐化（如四层架构约束、Lombok 规范）
- ⚠️ **SHOULD**：大多数情况应遵守，但特殊场景允许有理由的例外（如新增模块时应参考模块文档）
- 💡 **MAY**：锦上添花，按需阅读（如线程上下文文档）

**在 AGENTS.md 和 docs/ 中的使用方式**：

- AGENTS.md 文档索引中，每条引用标注强度，让 AI Agent 和开发者快速判断优先级
- conventions/ 文档中，每条规则标注强度，让 Code Review 有明确的判断标准
- modules/ 文档本身全是 💡 MAY 级别（按需查阅），但 AGENTS.md 中根据模块重要性区分引用强度

### 设计考量与权衡

| 权衡点 | 选择 | 理由 |
|--------|------|------|
| 文档数量多（25 个文件）vs 少 | 多 | 每个文档职责内聚，代码变更时只需更新对应文档；若合并为大文件，任何变更都需重新审阅整个大文件 |
| AGENTS.md 精简 vs 完整 | 精简 | AI 性能优先，详细内容委托给 docs/；硬性约束（⛔ MUST）保留内联 |
| 三轨 vs 两轨 | 三轨 | Intent 和 Contract 的时间维度不同（冻结 vs 跟随），Contract 和 Constraint 的方向维度不同（代码→文档 vs 文档→代码） |
| RFC 2119 三级 vs 四级 | 三级 | 经过 25+ 年验证，超过 3 级增加认知负担无有效性证据 |
| 模板固定 vs 灵活 | 双层（固定外壳 + 灵活内容） | 完全固定降低表达力，完全灵活失去模板意义 |
| modules/ 业务技术混合 vs 分开 | 混合 | 读者使用场景一致（了解模块功能），通过 README 分类表区分即可 |
| Mermaid 图 vs 纯文字 | Mermaid | 架构和设计用图比文字更直观，且 Mermaid 是纯文本格式可直接在 Markdown 中维护 |

## 文档治理

### 维护职责分工

文档维护操作按确定性程度分为三级：

#### 🤖 确定性维护（AI 可自主完成）

内容可从代码或配置直接推断，无需人类判断：

| 触发场景 | 操作 | 验证方式 |
|---------|------|---------|
| 代码新增公开方法 | 更新对应 modules/ 文档的 API 参考章节 | 方法签名与文档一致 |
| 代码新增配置项 | 更新对应 modules/ 文档的配置参考章节 | 配置项名称和默认值与代码一致 |
| 代码修改方法签名 | 更新对应 modules/ 文档的 API 参考章节 | 签名完全匹配 |
| 依赖版本变更 | 更新对应 architecture/ 文档的技术栈表格 | 版本号与 pom.xml 一致 |
| 新增 Maven 模块 | 按 modules/ 模板创建文档，更新 AGENTS.md 索引 | 模板结构完整，索引链接有效 |
| 文档中的代码示例过期 | 更新代码示例以匹配当前 API | 示例可编译运行 |

#### 🤖👤 半确定性维护（AI 提议，人类确认）

内容需要理解业务上下文或设计意图，AI 提议修改方案，人类确认后执行：

| 触发场景 | 操作 | 验证方式 |
|---------|------|---------|
| 业务场景变化 | 更新对应 modules/ 文档的业务场景描述 | 人类确认场景准确 |
| 新增架构决策 | 更新对应 architecture/ 文档，增加设计考量 | 人类确认设计理由充分 |
| 规则变更 | 更新对应 conventions/ 文档，AGENTS.md 约束摘要 | 团队讨论后确认 |
| 文档间交叉引用变化 | 更新"相关文档"章节的链接 | 人类确认关联关系正确 |
| 文档结构重组 | 调整章节顺序或合并拆分章节 | 人类确认结构合理性 |

#### 👤 创造性维护（人类主导）

内容需要原创性思考或团队共识，人类主导，AI 辅助：

| 触发场景 | 操作 | 验证方式 |
|---------|------|---------|
| 架构级变更（新增层次、模式变更） | 更新对应 architecture/ 文档 | 架构评审通过 |
| 新增编码规范 | 创建 conventions/ 文档 | 团队讨论后采纳 |
| 文档体系自身变更 | 更新 docs/README.md | 团队讨论后采纳 |
| 新增设计模式说明 | 更新 design-patterns.md | 设计评审通过 |

### 文档与代码对齐机制

三轨体系的核心挑战是如何保持文档与代码的一致性。按轨道分别说明对齐策略：

#### Contract 轨对齐

```
代码变更 → 触发对应的 modules/ 文档更新 → 更新 architecture/ 图表（如涉及架构变更）
```

- **对齐流程**：代码变更完成后，立即检查对应 modules/ 文档的 API 参考和配置参考章节
- **触发场景**：新增/修改/删除公开方法、新增/修改配置项、模块间依赖变化
- **漂移检测**：对比模块文档的 API 参考章节与实际代码的方法签名列表
- **自动化方向**：未来可引入编译期注解处理器自动生成 API 参考章节骨架

#### Constraint 轨对齐

```
规则变更 → 更新 conventions/ 文档 → 新代码必须遵守 → ArchUnit / Code Review 守护
```

- **对齐流程**：规则变更先更新 conventions/ 文档，再由 ArchUnit 测试和 Code Review 守护
- **触发场景**：新增/修改编码规则、依赖约束变化、架构约束变化
- **漂移检测**：ArchUnit 测试失败 = Constraint 轨文档与代码不一致
- **自动化方向**：ArchUnit 已守护四层架构约束；未来可扩展至更多规则

#### Intent 轨对齐

```
设计意图 → OpenSpec spec.md（冻结）→ 实现可能偏离 → 新的变更流程处理偏差
```

- **对齐流程**：Intent 文档不跟随代码变化——这是特性不是缺陷
- **触发场景**：实现偏离设计意图
- **漂移检测**：人工对比 spec.md 与实际实现的差异
- **自动化方向**：通过新的 OpenSpec 变更流程记录偏差原因，形成决策时间线

|---------|---------|------|--------|---------|
| 代码变更（新增方法） | 每次代码变更 | 更新对应 modules/ 文档的 API 参考章节 | 🤖 | 方法签名与文档一致 |
| 代码变更（新增配置项） | 每次配置变更 | 更新对应 modules/ 文档的配置参考章节 | 🤖 | 配置项名称和默认值与代码一致 |

| 架构变更（新增层/模块） | 重大架构调整 | 更新对应 architecture/ 文档 + AGENTS.md 文档索引 | 👤 | 架构评审通过 |

### 反模式警示

以下是本体系刻意避免的四类反模式：

#### 文档结构反模式

| 反模式 | 为什么不好 | 本体系的做法 |
|--------|----------|------------|
| AGENTS.md 塞入所有内容 | 过长降低 AI 性能，与 docs/ 重复 | AGENTS.md 只放约束摘要 + 引用索引 |
| 按"文档类型"组织 docs/ | 信息分散，一个模块的信息散布在多个文件 | 按"模块"组织，一个模块一个文件 |
| 无引用强度标注 | 开发者不知道哪些文档是必须读的 | RFC 2119 三级体系：⛔/⚠️/💡 |

#### AI 维护反模式

| 反模式 | 为什么不好 | 本体系的做法 |
|--------|----------|------------|
| AI 一次性重写整个文档 | 丢失人类审阅的增量变化，引入未预期的不一致 | 按章节定位精准编辑，保持未变更章节不动 |
| AI 修改 Intent 轨文档 | 破坏冻结约束，丢失原始设计意图 | Intent 轨只通过新的 OpenSpec 变更流程修改 |
| AI 全文加载大文档 | 浪费上下文窗口，降低输出质量 | 使用 `md-sections` 工具按需加载章节 |

#### 维护遗漏反模式

| 反模式 | 为什么不好 | 本体系的做法 |
|--------|----------|------------|
| 代码变更后不更新文档 | 文档与代码漂移，失去参考价值 | Contract 轨标签提醒 + 维护策略表格覆盖常见场景 |
| 新增模块不创建文档 | 模块无文档可查，信息散落在代码注释中 | modules/ 模板 + AGENTS.md 索引更新为固定流程 |
| 精确版本号写在多处 | 版本升级时需同步更新多处 | 精确版本仅在 architecture/ 文档中维护 |

#### 内容膨胀反模式

| 反模式 | 为什么不好 | 本体系的做法 |
|--------|----------|------------|
| 在文档中复制其他文档的内容 | 两份文档同一内容，更新时需同步多处 | 需要引用其他文档的内容时，不复制，用链接 |
| 文档内容超出其职责范围 | 职责模糊化，读者无法快速定位 | 每份文档一句话概括职责，扩展内容用"相关文档"链接 |
| 章节层级过深（L4+） | 读者难以定位，维护者难以保持结构 | 章节不超过 L3，L3 用于细分 |

### 可演进性设计

本文档体系在设计时考虑了以下演进方向，每个方向包含触发条件和实施步骤：

#### 新增模块

- **触发条件**：新增 Maven 模块（在 clients/ 目录下新增客户端模块，或在 app/ 下新增业务模块）
- **实施步骤**：① 在 modules/ 下按模板创建文档 → ② 填充内容 → ③ 在 AGENTS.md 文档索引中添加引用 → ④ 在 modules/README.md 文档列表中添加条目

#### 新增规范

- **触发条件**：团队讨论后新增编码规范或约束
- **实施步骤**：① 在 conventions/ 下按模板创建文档 → ② 填充规则和示例 → ③ 在 AGENTS.md 约束摘要中添加 ⛔ MUST 级别规则 → ④ 在 AGENTS.md 文档索引中添加引用

#### 新增架构文档

- **触发条件**：出现新的架构级关注点（如引入新的设计模式、新的部署架构）
- **实施步骤**：① 在 architecture/ 下创建新文件 → ② 回答一个新的架构问题 → ③ 按 architecture/README.md 模板填充 → ④ 在 AGENTS.md 文档索引中添加引用

#### 引入 ADR 体系

- **触发条件**：团队需要记录轻量级架构决策（OpenSpec 的 Intent 轨更重，适合完整变更流程）
- **实施步骤**：① 在 docs/ 下新增 `decisions/` 目录 → ② 定义 ADR 模板（上下文/决策/状态/后果）→ ③ 与 OpenSpec Intent 轨互补使用

#### 搭建文档站点

- **触发条件**：团队规模扩大，需要在线文档浏览和搜索
- **实施步骤**：① 选择静态站点生成器（VitePress / Docusaurus）→ ② 配置 docs/ 目录为内容源 → ③ 配置 Mermaid 插件支持图表渲染

#### 引入 CI 文档新鲜度检查

- **触发条件**：文档与代码漂移频繁发生，需自动化检测
- **实施步骤**：① 编写 CI 脚本检测 docs/ 文件的最后修改时间与对应源码文件 → ② 设置阈值（如超过 30 天未更新）→ ③ 自动创建 Issue 提醒

#### 演进决策树

面对一个文档变更，按以下路径选择演进方式：

```
代码变了？
├── 是 → 变更涉及哪个层级？
│   ├── 方法/配置级 → 更新对应 modules/ 文档（🤖）
│   ├── 模块级 → 更新对应 modules/ 文档 + architecture/ 图表（🤖👤）
│   └── 架构级 → 更新对应 architecture/ 文档 + 评估是否需要新文档（👤）
└── 否 → 是什么类型的变更？
    ├── 新增约束 → 创建/更新 conventions/ 文档（🤖👤）
    ├── 新增关注点 → 评估是否需要新增文档（👤）
    └── 体系调整 → 更新 docs/README.md + AGENTS.md（👤）
```

## 与 AI Agent 协同

### AGENTS.md 的定位

AGENTS.md 是 AI 编码助手的"入口文件"，提供：

1. 精简的硬性约束摘要（⛔ MUST 级别的编码规则）
2. 带引用强度的文档索引（让 AI 知道哪些文档重要、哪些可选）
3. 不包含可从代码推断的内容（项目结构索引、精确版本号、JVM 详表）

**为什么删除"项目结构索引"？** ETH Zurich 的实验表明，AI Agent 可以通过文件系统直接获取项目结构，在 AGENTS.md 中重复列出反而增加噪音、降低性能。

**为什么删除精确版本号？** AI Agent 可以直接读取 pom.xml 获取精确版本号。AGENTS.md 只保留大版本号（Spring Boot 4.x 而非 4.0.2），降低维护成本的同时不影响 AI 的代码生成能力。

### docs/ 对 AI 的价值

当 AI 需要了解某个模块的详细设计时，可以通过 AGENTS.md 的引用索引定位到对应的 modules/ 文档，获取准确的类名、方法签名和配置项。

当 AI 需要理解架构决策时，可以通过 architecture/ 文档获取 C4 图、时序图和设计理由。

当 AI 需要遵守编码规范时，可以通过 conventions/ 文档获取精确规则和示例代码。

### AI 可自主维护的文档范围（🤖 确定性维护）

以下维护操作 AI 可自主执行，无需人类确认：

- 代码新增/修改公开方法时，更新对应 modules/ 文档的 API 参考章节
- 代码新增/修改配置项时，更新对应 modules/ 文档的配置参考章节
- 依赖版本变更时，更新对应 architecture/ 文档的技术栈表格
- 新增 Maven 模块时，按 modules/ 模板创建文档

### AI 需人类确认的维护范围（🤖👤 半确定性维护）

以下维护操作 AI 提议修改方案，人类确认后执行：

- 业务场景变化时，更新对应 modules/ 文档的业务场景描述
- 新增架构决策时，更新对应 architecture/ 文档
- 规则变更时，更新对应 conventions/ 文档
- 文档间交叉引用变化时，更新"相关文档"章节

### 文档加载规则

AI Agent 加载 docs/ 下的文档时，**⛔ 绝对禁止⛔ 直接读取全文**。必须使用 `scripts/md-sections` 工具：

1. 先获取章节结构：`scripts/md-sections <file>`
2. 按需加载章节：`scripts/md-sections <file> "标题1" ["标题2" ...]`

详见 AGENTS.md 中的"文档引用规则（分层披露原则）"。
