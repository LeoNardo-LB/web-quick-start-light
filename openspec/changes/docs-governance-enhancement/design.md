## Context

项目文档体系经过 `docs-system-rebuild` 变更已完成三轨文档体系的搭建（28 个文件），当前存在以下结构性问题：

**当前 docs/README.md 的结构问题**：15 个 `###`（L3）子章节全部堆在 `## 文档系统设计` 之下，维护策略、反模式警示、对齐机制等治理内容与设计初衷、参考资料等设计内容混在一起。读者无法区分"体系怎么设计的"和"我该怎么维护这个体系"。

**当前 AGENTS.md 的缺失**：没有引用文档治理相关章节，AI Agent 不知道哪些文档是自己该维护的。

**当前 modules/ 模板不完整**：固定层只有 6 个章节（概述→业务场景→技术设计→API参考→配置参考→使用指南），缺少"相关文档"和"变更历史"，导致 12/12 文件全部缺少这两个章节。

**当前文档深度不足**：docs/ 下的文档面向人类，应有足够的图表、场景描述和设计考量，但当前许多文档的业务场景只有 4-7 行，conventions/ 缺少"设计考量"和"常见违规场景"。

**三轨标签不可见**：`md-sections` 脚本读取标题时看不到文档属于哪条轨道。

**约束**：
- 本次变更仅涉及文档文件，不涉及任何代码变更
- 必须遵循"索引具体，规则泛化"原则
- 文档间的职责边界不能模糊化（扩展内容时不能侵犯其他文档的职责）

## Goals / Non-Goals

**Goals:**

1. 将文档治理维度从设计子章节中独立出来，成为 docs/README.md 的一等公民（L2 章节）
2. 建立人类 vs AI 的三级维护职责分类（确定性/半确定性/创造性），让 AI Agent 明确知道哪些文档自己可以自主维护
3. 建立宪法级文档编写原则（7 个维度），作为所有文档的编写指南
4. 在 AGENTS.md 中直接引用治理章节，让 AI Agent 在会话入口就能获取维护职责
5. 升级 modules/ 模板固定层，补全缺失的"相关文档"和"变更历史"章节
6. 在文档标题上体现三轨标签，使 `md-sections` 脚本读取时可见
7. 扩展全部 docs/ 文档的内容深度（业务场景、设计考量、常见违规、多场景示例等）
8. 同步优化各 README.md 模板说明，确保模板能指导扩展后的内容编写

**Non-Goals:**

- 不改变三轨文档体系的核心架构（三轨分类标准不变）
- 不新增文档文件（不新增 architecture/conventions/modules 下的文件）
- 不改变 OpenSpec Intent 轨的任何内容
- 不改变 `scripts/md-sections` 脚本的行为
- 不做代码层面的变更（模板、校验脚本等均不变）
- 不进行文档的自动化检查或 CI 集成（这是未来的演进方向，本次不实现）

## Decisions

### D1: docs/README.md 结构重组策略

**选择**：将治理相关内容从 `## 文档系统设计` 中剥离，独立为 `## 文档治理` L2 章节。

**替代方案**：
- A) 保持 L3 不变，只扩展内容 → 治理内容仍然被设计内容淹没，不符合"提升优先级"的需求
- B) 将每个治理子章节都提升为 L2 → 太碎片化，4-5 个新 L2 章节会让目录膨胀
- C) 合并为一个 `## 文档治理` L2 章节（本方案）→ 聚合相关内容，目录结构清晰

**最终结构**：
```
# 文档系统
  📋 目录                    L2
  三轨文档体系              L2  （保持）
  ★ 文档编写原则            L2  （🆕 新增，宪法级）
  文档模板                  L2  （保持）
  文档导航                  L2  （保持）
  引用强度说明              L2  （保持）
  文档覆盖度                L2  （保持）
  文档系统设计              L2  （精简，移除治理内容）
    设计初衷
    设计理念
    整体架构
    OpenSpec Intent 轨
    三轨分类标准
    docs/ 目录的组织方式
    模板系统的设计哲学
    引用强度体系
    设计考量与权衡
    与业界实践的关系
    参考资料
  ★ 文档治理                L2  （🆕 从子章节提升独立）
    ★ 维护职责分工           L3  （🆕 新增）
    文档与代码对齐机制       L3  （提升+扩展）
    维护策略                 L3  （提升+大幅扩展）
    反模式警示               L3  （提升+大幅扩展）
    可演进性设计             L3  （提升+大幅扩展）
  ★ 与 AI Agent 协同        L2  （从 L3 提升+大幅扩展）
  相关文档                  L2  （保持）
```

**理由**：治理内容与设计内容的读者心智模型不同——设计内容回答"体系为什么这样设计"，治理内容回答"我怎么维护这个体系"。独立后读者可以按需阅读。

### D2: 维护职责三级分类模型

**选择**：按"确定性程度"分三级，而非按"轨"或按"文档类型"分。

```
🤖 确定性维护（AI 可自主完成）
   内容可从代码直接推断，无需创造性判断
   触发：代码变更后自动触发
   示例：新增方法 → 更新 API 参考章节

🤖👤 半确定性维护（AI 提议，人类确认）
   需要一定判断，AI 生成初稿，人类 Review
   触发：代码变更后 AI 提议，或人类要求
   示例：类图变更 → 更新技术设计章节

👤 创造性维护（人类主导）
   需要深度思考和设计判断
   触发：架构级决策或体系级变更
   示例：新增架构层 → 更新 architecture/ 文档
```

**替代方案**：
- A) 按轨分（Contract=AI，Constraint=人类）→ 太粗糙，AI 其实也可以维护部分 Constraint 内容
- B) 按文件类型分（modules/=AI，architecture/=人类）→ 不准确，modules/ 的技术设计也需要人类判断
- C) 按确定性程度分三级（本方案）→ 最精确，反映了维护操作的本质差异

**理由**：维护职责的划分标准不应该是"文档是什么类型"，而应该是"这个维护操作需要多少判断力"。同样的 modules/ 文档，更新 API 签名是确定性操作（AI 可自主），但更新类图是半确定性操作（需要人类确认）。

### D3: 文档编写原则的 7 维度设计

**选择**：7 个维度，每个维度有明确的判断标准。

1. **职责清晰** — 每份文档一句话概括职责，扩展内容前先问"这属于我的职责吗？"
2. **简要精准** — 表格优于段落，列表优于散文，代码示例优于文字描述
3. **层次分明** — 固定层不修改，半固定层按需填充，章节标题即信息
4. **逻辑自洽** — 文档内不矛盾，文档间不重复，用链接不用复制，矛盾时以真相源为准
5. **科学有据** — 设计决策有理由，引用业界标准时标注来源
6. **双范式写作** — 面向 AI（精确可执行、步骤可闭环）vs 面向人类（构图+说明+场景驱动）
7. **索引具体，规则泛化** — 索引精确到文件，规则面向未来不绑死文件名

**替代方案**：
- A) 只保留当前的设计理念（真相源唯一+职责内聚+读者导向+最小维护成本）→ 太抽象，无法指导具体编写
- B) 每个文档类型单独制定原则 → 太碎片化，增加维护成本
- C) 统一的 7 维度原则（本方案）→ 通用且可操作

**理由**：设计理念回答"体系为什么这样设计"，编写原则回答"我写文档时怎么判断写得对不对"。两者互补不重复。

### D4: 三轨标签在标题上的呈现方式

**选择**：在一级标题后以独立句子说明轨道归属和维护方向，而非嵌入标题文本。

```markdown
# 认证模块

> 🟢 Contract 轨 — 100% 反映代码现状，代码变更时必须同步更新
```

**替代方案**：
- A) 嵌入标题：`# 🟢 认证模块` → 标题变长，且 `md-sections` 输出的 `title` 字段会带上 emoji
- B) 放在正文第一段 → `md-sections` 只读标题，脚本层面看不到
- C) 用 blockquote 独立一行放在标题下方（本方案）→ 标题保持简洁，blockquote 内容在视觉上与正文区分

**理由**：`md-sections` 脚本当前只解析 `#` 标题行，不解析 blockquote。但用户希望脚本读取标题时就能看到轨道信息。因此实际方案调整为**将轨道标签嵌入标题本身**：

```markdown
# 认证模块（🟢 Contract 轨）
```

这样 `md-sections` 的 JSON 输出中 `title` 字段直接包含轨道信息。但考虑到 emoji 在 JSON 中可能有兼容性问题，最终选择用文字标签：

```markdown
# 认证模块 — Contract 轨
```

**最终决定**：在标题中追加轨道标记（括号形式），紧跟一级标题。标题下方 blockquote 不再重复轨道标签，改为补充说明维护方向。

### D5: modules/ 模板固定层扩展

**选择**：在 6 章节固定层后增加"相关文档"和"变更历史"，变为 8 章节固定层。

```
固定层（不可修改）：
  概述 → 业务场景 → 技术设计 → API 参考 → 配置参考 → 使用指南 → 相关文档 → 变更历史
```

**理由**：architecture/ 的 6 个文件全部有"相关文档"和"变更历史"章节，modules/ 的 12 个文件全部没有。这是模板设计的遗漏而非有意为之。两个章节对文档网络的可导航性和可追溯性至关重要。

### D6: 实施分 Phase 策略

**选择**：4 个 Phase，按依赖关系排序。

- **Phase 1**：docs/README.md 结构性重构（所有后续工作的基础）
- **Phase 2**：AGENTS.md 更新（依赖 Phase 1 的章节锚点）
- **Phase 3**：模板升级 + 三轨标签统一（依赖 Phase 1 的编写原则和 Phase 2 的 AGENTS.md 引用）
- **Phase 4**：各文档内容扩展（依赖 Phase 3 的模板更新，且工作量最大）

**理由**：Phase 1-3 是结构性变更（骨架），Phase 4 是内容性变更（血肉）。先搭骨架再填内容，避免内容扩展后又要调整结构。

### D7: 内容扩展的"纵向深挖+横向关联"策略

**选择**：不横向扩展职责范围，而是纵向深挖现有职责范围 + 横向建立文档间关联。

**纵向深挖方向**（在当前职责范围内增加深度）：
- modules/：业务场景从列表扩展为场景描述，使用指南增加多场景代码示例，技术设计增加设计考量
- conventions/：每条规则增加"设计理由"和"常见违规场景"
- architecture/：适当增加系统边界说明和完整列表

**横向关联方向**（建立文档间网络）：
- 所有 modules/ 文档补全"相关文档"章节，链接到相关的 architecture/ 和 conventions/ 文档
- conventions/ 文档增加规则间的交叉引用
- 遵循原则："需要引用其他文档的内容时，不复制，用链接"

**替代方案**：
- A) 为每个模块增加"设计意图"章节 → 这是 Intent 轨的职责（OpenSpec），Contract 轨不应包含
- B) 将 conventions/ 的规则详细说明移入 modules/ → 规则是约束，应该集中在 conventions/，modules/ 只引用

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| docs/README.md 行数大幅增加 | 当前 434 行，扩展后可能 700+ 行 | 这是体系说明文档，行数增长合理；且 `md-sections` 支持按章节读取，不影响 AI 使用效率 |
| 全部文档同时扩展可能引入不一致 | 22 个文件同时修改 | Phase 4 分多个 Task，每个 Task 聚焦一个目录（modules/ → conventions/ → architecture/） |
| 三轨标签加在标题上会改变 `md-sections` 输出 | 所有使用 `md-sections` 的脚本/流程需要适应 | 标签格式统一为括号形式，不影响标题的语义 |
| 维护职责分工过于理想化 | 实际场景中 AI 和人类的维护边界可能模糊 | 三级分类是指导而非硬性约束，允许根据实际调整 |
| 内容扩展可能模糊文档间边界 | 扩展后文档职责可能重叠 | 每次扩展前检查"这属于我的职责吗？"，用链接代替内容复制 |
